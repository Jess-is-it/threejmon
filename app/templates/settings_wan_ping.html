Target Latency

How Target Latency works
{% set enabled_targets = settings.general.targets | selectattr('enabled') | list %}
How we collect ping (per ISP)
  • Ping is executed inside the MikroTik using RouterOS API /tool/ping (not from this server).
  • We set src-address so results are ISP-scoped. The saved Local IP must be an exact local interface IP on that router.
  • We do not retry without src-address (to keep results ISP-scoped). No reply is stored as a timeout sample.
  • By default, each poll runs against all enabled targets. You can enable target rotation in WAN Settings → Interval → Target Latency Polling if you need lower router load.
  • If a ping gets a Request timeout, we store it as a failed sample (no RTT). Charts show a gap and tooltips display Timeout.
  • The Worst value on each target card is the max RTT in the selected window (for the currently enabled ISP filters).
  • Charts show max RTT per bucket (≤1D: 5m, ≤7D: 30m, >7D: 2h). Tooltips use Asia/Manila (PHT).

Target Latency Polling

Controls how Target Latency charts are sampled and stored. Aggressive settings can increase router load and database size.
Guide
How often to ping targets for Target Latency charts (separate from WAN status).
Off (default): ping all enabled targets every poll. On: rotate targets per poll using the field below.
0 = auto (all selected targets in parallel). Set 1 for sequential. Higher values increase concurrency and router load.
Used only when rotation is enabled. 1 = rotate lightly, higher values = more targets per poll, 0 = disable Target Latency sampling.
Timeout per ICMP attempt inside RouterOS /tool/ping.
How many pings to run per target sample. Higher values are heavier but may be more stable.
How polling works
  • Every Target Latency Interval, each enabled ISP pings either all enabled targets (rotation OFF) or Targets per ISP per poll (rotation ON).
  • Ping execution per ISP is parallelized by Parallel workers per ISP (0 = auto/all selected targets).
  • Pings run inside the MikroTik via RouterOS API /tool/ping with src-address = the ISP Local IP (so results are ISP-scoped).
  • Estimated revisit time per target (rotation ON) ≈ ceil(enabled_targets / targets_per_poll) × target_interval.
  • Estimated router load ≈ ISPs_on_router × targets_per_poll × ping_count pings per poll.
  • If you formatted the WAN database, Target Latency history was cleared too. Charts will show n/a until new samples arrive.
