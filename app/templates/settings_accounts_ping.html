Classification
{% set patch_ctx = accounts_ping_classification_patch or {} %} {% set patch_preview = patch_ctx.preview or {} %}
How Classification works in plain language: Accounts Ping uses two kinds of judgment at the same time. First, it checks the latest live ping result to know whether the account is currently up or down. Second, it looks at the selected page window, such as 1h, 6h, 24h, or 7d, and calculates how often that account failed during that period. The Fail % is based on that selected window, not only on the latest ping.
This creates three practical bands: Stable when the account is up and its Fail % stays at or below Stable Fail %; Monitor when the account is still up but its Fail % rises above Stable Fail %; and Issue-level Monitor / Down when the account is currently down or its Fail % reaches Issue Fail %, or its latest loss/latency crosses the issue thresholds.
About the thresholds that may look similar: Stable Fail % and Issue Fail % are not two different measurements. They are two levels of the same Fail % score. Stable Fail % is the warning line. Issue Fail % is the more serious line. So they are related, but not redundant.
About Down Loss % vs Issue Loss %: these also work differently. Down Loss % is used by the live checker to decide whether the account should be treated as currently down for that ping cycle. Issue Loss % is the warning or issue threshold shown in the UI when the account is still being evaluated for health quality. In most deployments Down Loss % stays at 100%, while Issue Loss % is lower so the portal can flag bad quality before full failure.
